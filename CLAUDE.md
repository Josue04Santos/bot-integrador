## Project Overview

**Bot Integrador DPL Construções** — A Telegram automation bot that batches queries to `@ReincidenciasBot` (an electrical network data bot), persists results in a database, and exports geographic files (KML + GPX + CSV) for field operators. Proprietary project for internal use at DPL Construções.

## Architecture

The application runs three concurrent tasks inside a single `asyncio.gather()` in `src/main.py`:

1. **aiogram bot** (`src/bot/`) — Telegram bot interface for operators. Handles `/start`, `/kml`, `/whoami`, `/status`, admin commands. Uses FSM (`QueryStates`) to collect batch codes from users.

2. **Worker loop** (`src/userbot/worker.py`) — Consumes `QueryQueue` (an in-memory `asyncio.Queue` singleton at `src/dispatcher/queue.py`). For each `QueueItem`, calls the userbot, updates DB, and sends results back to the user.

3. **Telethon userbot** (`src/userbot/client.py`) — Logs in as a real Telegram user account to interact with `@ReincidenciasBot`. Uses a conversational flow: sends `/PTE` or `/EQP`, waits for prompt, sends the code, collects response. **Session is persisted in the `telethon_sessions` table in Postgres** (not a `.session` file).

### Query Lifecycle

```
User sends codes → query.py handler →
  Creates QueryBatch + NetworkQuery rows (status: pending) →
  Enqueues QueueItem → worker picks it up →
  UserbotClient.query_poste() / query_equipamento() →
  Updates DB (status: received/timeout/error) →
  Notifies user (individual result) →
  If batch complete: sends summary + download button
```

### Export Pipeline (`src/exporters/__init__.py:generate_bundle`)

When the user clicks "Download" or runs `/kml <batch_id>`:
1. Loads all `NetworkQuery` rows for the batch
2. Parses raw responses → `PosteData` or `EquipamentoData` (branched by `query_type`)
3. Runs **OR-Tools TSP** (`RouteOptimizer`) on postes with coords for optimal ordering
4. Calls **OSRM** (`src/services/osrm_client.py`) for real road geometry
5. Builds KML + GPX (postes) + GPX (equipamentos) + 2 CSVs + invalids TXT
6. Returns `ExportBundle` with all file bytes

The `adapter.py` module decouples `PosteData` (exporters domain) from `RoutePoint` (services domain).

## Key Conventions

### Module execution
Always run as `python -m src.main`, not `python src/main.py`. The `conftest.py` adds the project root to `sys.path` for pytest, but the module form is required for production.

### Settings
Single settings instance via `from src.config import settings`. All keys are **snake_case** (`settings.telegram_bot_token`, not `settings.TELEGRAM_BOT_TOKEN`). Loaded from `.env` by `pydantic-settings`. Switching SQLite ↔ PostgreSQL: set `database_backend=postgres` in `.env`.

### Database
- `db = DatabaseManager()` singleton in `src/database/connection.py`
- Always use `async with db.session() as session:` — auto-commits on success, rolls back on exception
- Primary keys are UUIDs generated by `uuid7()` from `src/database/types.py`
- Models are SQLAlchemy 2.0 with `Mapped` typed columns
- Schema: `AuthorizedUser` → `QueryBatch` (1:N) → `NetworkQuery` (1:N) → `Meter`

### Bot handlers
Handlers are aiogram `Router` instances, registered in order in `src/bot/handlers/__init__.py:register_handlers()`. Order matters — `admin_router` is first, `help_router` is last (catch-all). Auth is enforced by `AuthMiddleware` (blocks users not in `authorized_users` table or with `active=False`). The middleware injects `auth_user_id`, `auth_user_role`, and `auth_user_full_name` into handler `data`.

### Parser design
Both `src/exporters/parser.py` (postes) and `src/exporters/parser_equipamento.py` (equipamentos) parse raw text responses from `@ReincidenciasBot`. They always return a data object (never raise) — check `.parse_error` to detect failures. Regex patterns are compiled at module level for performance. The raw response is always stored in `NetworkQuery.raw_response` to allow re-parsing if the parser changes.

### Logging
Uses `structlog` throughout. Get a logger with `from src.utils.logger import get_logger; logger = get_logger(__name__)`. In development: colored console output. In production: JSON.

### Query types
`query_type` values in the DB: `"poste"` and `"instalacao"` (not `"equipamento"`). The userbot sends `/PTE` for postes and `/EQP` for equipamentos. This naming inconsistency (`instalacao` in DB vs `equipamento` in UI) is intentional — it matches the upstream bot's terminology.

## Environment Setup

Copy `.env.example` to `.env`. Required variables:
- `TELEGRAM_BOT_TOKEN` — from @BotFather
- `TELEGRAM_API_ID`, `TELEGRAM_API_HASH`, `TELEGRAM_PHONE` — from [my.telegram.org](https://my.telegram.org)

Default database is SQLite at `data/bot_integrador.db`. Switch to Postgres by setting `database_backend=postgres` plus the `POSTGRES_*` vars.

The Telethon session requires an **interactive first login** (code + optional 2FA). After that, the session string is saved to the `telethon_sessions` table and reused automatically.

## First-time User Authorization

Users must be manually inserted into `authorized_users` before they can use the bot (there's no `/grant` command yet — it's in the roadmap):

```python
from src.database.connection import db
from src.database.models import AuthorizedUser

async with db.session() as s:
    s.add(AuthorizedUser(tg_id=123456789, full_name="Name", role="admin"))
    await s.commit()
```

## Infrastructure

- **Deploy**: Docker Compose in `deploy/docker/` (Dockerfile + compose)
- **Nginx**: config in `deploy/nginx/`
- **OSRM**: uses the public `router.project-osrm.org` (free, fair-use). Rate-limited to 1 req/s. For heavy production use, self-host OSRM.
- **Export files**: generated in-memory as bytes (not written to disk in the export pipeline)
